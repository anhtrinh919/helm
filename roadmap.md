# Roadmap — Helm

> **Full re-cut (2026-06-09).** Helm is repositioned as the no-code app studio for non-developers — the Replit/Bolt/Lovable alternative that runs on your own Claude subscription. The roadmap below is re-derived from hands-on competitive research (`research/replit.md`, `research/bolt.md`, `research/lovable.md`) and supersedes the prior Phase 1–4 plan. The shipped code becomes Phase 0 (the foundation); every forward phase is re-derived from the four differentiators (guided journey, Claude-Max economics, cards+cockpit, preview+cockpit) plus the newly-in-scope shipping and local-backend capabilities.

## Phases

0. **Phase 0 — Foundation** ✓ SHIPPED *(consolidates the prior Phases 1–3 and the Phase 4 data-layer groundwork)*: The no-code cockpit engine that everything else builds on. The desktop shell; scoped named sessions wired to the live Claude Agent SDK; the project board (features/bugs/decisions); the embedded Live Preview (the real local app running inside Helm); the Point-and-Fix overlay (click an element, spawn a fix session with visual context); real-data Decisions/Progress/Docs; durable background sessions; parallel sessions. Also shipped but not yet surfaced: the data-layer for project modes, the build journey/rail cursor, the For-Later shelf, session-stop, wizard-state persistence, legacy-project mapping, and import-folder scanning. This layer is design-independent and stable; forward phases surface and extend it.

1. **Phase 1 — The Cockpit & the Journey** ← NEXT: The identity phase. A total visual redesign (calm, sleek, modern, guiding — bold where it counts) applied across every screen, plus the two-door front door and the **guided Build journey** surfaced on top of the shipped rail data-layer. This is differentiators #3 (cards+cockpit, not chat) and #4 (preview+cockpit, not code) made real: the cockpit wraps the live app in feature/outcome context, the build journey gives the structured path that one-shot competitors lack, the For-Later shelf and journey checkpoints become visible, and project management (rename/delete/organize, truthful Live status, consistent nav) rides along. Point-and-fix is sharpened to match or beat Lovable's visual edit. Ends dogfoodable in the real Electron app: a non-dev can start a project at the front door, move through the journey, and live in the cockpit. *(This subsumes the in-flight "Build & Iterate" work, now reframed as the cockpit identity rather than a standalone feature.)*

2. **Phase 2 — Ship It**: One-click publish. The locally-running app goes live on a real, shareable URL on the user's own Cloudflare domain (`<slug>.ta-infra.uk`) via a tunnel — the app keeps running on the user's machine; the world reaches it through the tunnel. A Publish/Ship panel shows the URL, who can reach it, and start/stop/status. Closes the single biggest gap vs the competitors (all three make "Publish" a core verb; Helm had none). Ends dogfoodable: a non-dev publishes a built app and opens its URL from another device.

3. **Phase 3 — Real Backend (local)**: Apps stop being static front-ends. Asking for accounts or saved data in plain language provisions a **local** auth + database + file storage and wires the app to them — no cloud account, no code shown. A plain-language Data Browser lets the user see what the app has stored (records, not SQL). This is the local-first answer to Lovable Cloud / Replit DB / Bolt+Supabase — without renting anyone's infrastructure. Ends dogfoodable: a non-dev adds a login and a saved-data feature and sees real records persist.

4. **Phase 4 — Stay Unstuck (error recovery + rollback)**: Trust hardening — the two things that make non-devs abandon a build. **Proactive error recovery:** Helm watches the running app for breaks (blank screen, build/runtime errors) and offers a one-click fix; the user never has to notice, find, or paste an error from a developer console — the universal non-dev gap that *none* of the competitors close. **Version history / rollback:** every change is a checkpoint with a plain-language label; the user previews any past state and reverts in one click (matching the competitors' strongest safety feature). Ends dogfoodable: a deliberately broken build is recovered without the user reading an error, and a bad change is rolled back.

5. **Phase 5 — Bring It In (import & continue)**: The migration wedge, hardened. Bring an app started in Lovable, Bolt, Replit (via their export/download), or any local web app, into Helm and continue iterating — Helm scans the folder, figures out how to run it, shows the live app, and treats it like any Helm project (cockpit, point-and-fix, ship, backend). Bolt already ships an "import from Lovable" path; Helm makes the destination the better home. Ends dogfoodable: a non-dev imports a real app exported from a competitor and ships it from Helm.

## Global Out of Scope

- Code editor or IDE panel — Helm never shows code to the user
- Terminal panel — all agent output is rendered as structured session feed, never raw terminal
- Exposing git, branches, diffs, or pull requests to the user — these are hidden scaffolding
- Charging money, a paid API path, a token meter, or any paid tier — free and open-source, bring-your-own-Claude-subscription; the AI runs on the user's existing Claude subscription at zero marginal cost
- Helm-hosted / multi-tenant SaaS, or a Helm-managed cloud backend — the app and its data run on the user's own machine; "publish" exposes the local app via the user's own Cloudflare domain, it does not move the app to Helm's servers
- User accounts *for Helm itself* — Helm is a local single-user desktop app (the apps it builds can have their own accounts; Helm does not)
- Any form of multi-user or team collaboration on Helm itself — single-user desktop app
- Auto-managed cloud databases / rented backend infrastructure — the provisioned backend is local-first (Phase 3)
