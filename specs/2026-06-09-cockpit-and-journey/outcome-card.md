# Phase 1 — The Cockpit & the Journey

## Phase goal
Give Helm its new identity: a calm, modern cockpit where a non-developer chooses how to start (build something new vs. iterate on an app), is guided step-by-step toward a working app, and always sees what each piece is *for* — never code, never a chat they get lost in.

## Grounding principle (condition of approval)
The guided journey is not an invented sequence — it mirrors the real build-stack practice Helm itself is built with (pressure-test the idea → scope what you want → plan it → build the look → build the function → review & try it), translated into plain, non-technical steps and checkpoints. Every detailed user flow in this phase must trace to a real stage of that practice, so "a guided build stack, not a one-shot" is true in substance, not just in framing.

## You'll be able to
1. Start a "Build something new" project at a clear two-door front door and move through a guided journey of ordered milestones — testing what was built at each checkpoint and continuing when ready — so you're never lost in a one-shot chat.
2. Work in a cockpit where every card shows, in plain language, what it's *for* (the outcome it delivers) right next to the live app — so you always know what's been built and whether it matches what you asked.
3. Click anything in the live app and fix it — including editing text right where it sits — so you correct things by pointing, not by describing.

## Also included
- A second, freeform door — "Iterate on an app → start fresh" — where your first request both sets up the app and builds the first feature in one step.
- The "bring an existing app" door is shown at the front door but says it's coming soon (the full import flow lands in a later phase).
- Mid-journey detours are handled for you: small fixes happen now, bigger or unrelated requests get parked on a visible "For later" shelf so nothing derails and nothing is lost — and an escape hatch lets you leave the guided path for the freeform board whenever you want.
- A celebration moment when the journey's last step is done ("Your app now does what you set out to build"), after which the journey collapses to a compact strip and the project opens up to free iteration.
- A subtle, always-present reassurance that Helm is "Running on your Claude subscription" — no usage limits, no second bill — where the competitors put a token meter or upgrade nag.
- Project housekeeping: rename, delete, and reorder your projects; a truthful "Live" status with a working stop control; and consistent back/home navigation.
- The whole thing wears the new calm/sleek/modern visual language, applied across every screen.

## Not in this phase
- Publishing or deploying an app to a URL (that's Phase 2).
- Giving the app real logins, a database, or saved data (that's Phase 3).
- Helm catching broken screens for you, and version rollback (that's Phase 4).
- The full "bring an existing app" import flow — only the door is shown this phase; importing competitor exports and tricky setups is Phase 5.
- Any new building *capability* — the engine that builds, previews, and fixes is unchanged; this phase is about how it's presented and guided.

## Success looks like
1. Opening Helm shows two clear doors; choosing "Build something new" and describing a goal produces a visible ordered list of milestones with a "step N of M" feel, each step stopping at a checkpoint you can test and then continue past — and parking a big mid-journey request makes it appear on the "For later" shelf instead of derailing the build.
2. On the cockpit board, every card visibly states what it's for in plain language (e.g. "lets a visitor book a slot"), shown alongside the live running app — with done work settling into quiet history.
3. Clicking an element in the live app lets you describe a fix or edit its text in place, and the change comes back applied to that exact element — no need to say where it is.
